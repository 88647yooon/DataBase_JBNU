# 블록체인-DB 연동 웹 포털 조회 시스템 구축 로드맵

본 문서는 AWS MySQL(데이터베이스)과 로컬 Geth 노드(블록체인)를 연동하여 물류 데이터의 무결성을 검증하는 '웹 포털 조회 시스템' 구축을 위한 단계별 태스크리스트입니다.

---

## 📋 단계별 태스크리스트

### 🛠1단계: 백엔드 웹 서버 API 환경 구축 (Flask 또는 FastAPI)
사용자가 브라우저에서 입력한 번호를 받아 처리할 웹 서버의 뼈대를 만드는 단계입니다.
- [ ] 파이썬 웹 프레임워크 라이브러리 설치 (`pip install fastapi uvicorn` 또는 `pip install flask`)
- [ ] 기본 웹 서버 파일(`app.py`) 생성 및 기본 포트(예: 8000 또는 5000) 개방
- [ ] 사용자가 입력한 화물 번호를 수신할 검증 API 엔드포인트 설계 (예: `/api/verify?item_id=1004`)

### 🗄2단계: 웹 서버 내부 - AWS MySQL 연동 및 데이터 조회
기존에 만든 DB 클래스를 웹 서버 안으로 가져와 실시간 데이터를 긁어오는 단계입니다.
- [ ] 기존 `DBManager` 클래스를 웹 서버 프로젝트 모듈로 이식
- [ ] 사용자가 전달한 `item_id`를 조건으로 조인(JOIN) 검색을 수행하는 SQL 쿼리 메서드 구현
- [ ] 조회된 데이터(투플 인스턴스 정보)를 해싱하기 좋은 표준 포맷(JSON 문자열 또는 바이트 스크립트)으로 변환하는 로직 작성

### 🔗 3단계: 웹 서버 내부 - 로컬 Geth 노드 연동 및 원본 해시 추출
블록체인에 영구 박제된 원본 데이터 영수증을 꺼내오는 단계입니다.
- [ ] 웹 서버 통합 스크립트에 `Web3.py` 라이브러리 연동 (`http://127.0.0.1:8545`)
- [ ] DB에서 가져온 현재 투플 데이터의 실시간 `Keccak256` 해시값 계산 로직 구현
- [ ] Geth 노드에서 해당 `item_id`에 매핑된 과거 원본 트랜잭션 데이터(또는 스마트 컨트랙트 저장값) 조회 함수 구현

### 💻 4단계: 프론트엔드 - 검색 포털 화면 및 결과 페이지 구현
사용자가 실제로 마주할 UI(조회창)와 검증 통과/실패 화면을 만드는 단계입니다.
- [ ] 메인 홈 화면 구현: 일반 택배사 조회창처럼 화물 고유 번호를 입력할 수 있는 간단한 HTML Form 생성
- [ ] 결과 화면(HTML Template) 설계: 웹 서버에서 판정한 결과에 따라 시각적 피드백 제공
  - **성공 시**: "무결성 검증 성공: 유통 경로가 원본과 일치합니다." (그린 라이트 효과)
  - **실패 시**: "무결성 검증 실패: 데이터 위변조가 감지되었습니다!" (레드 경고 효과 및 상세 경로 대조 테이블 표시)

### 🧪 5단계: 통합 테스트 및 위변조 시뮬레이션
시스템이 정상적으로 해킹을 잡아내는지 강제로 테스트해 보는 마지막 단계입니다.
- [ ] 정상 시나리오 테스트: AWS MySQL에 정상 투플을 넣고 포털에 검색했을 때 정상 판정이 뜨는지 확인
- [ ] 위변조 시나리오 테스트: AWS MySQL 워크벤치 등으로 접속하여 특정 투플의 데이터(예: 이동 경로 중 하나)를 강제로 수정한 후, 포털에서 검색했을 때 **"위변조 감지"** 경고가 완벽하게 터지는지 확인

---

## 🏛시스템 데이터 흐름 아키텍처

```
[스마트폰 / 웹 브라우저] (화물 번호 입력: item_id=1004)
          │
          ▼
[파이썬 웹 서버 (API)]
          │
          ├─► [AWS MySQL] 현재 유통 경로 데이터 조회 (실시간 해시 생성)
          │
          └─► [Geth 로컬 노드] 블록체인 원본 해시 조회
          │
          ▼
[두 해시값 비교 (Keccak256)] ──► 일치 여부에 따라 최종 결과 화면(HTML) 반환
```
